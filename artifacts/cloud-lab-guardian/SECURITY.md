# Security Policy

## Overview

Cloud Lab Guardian is an educational tool that generates safe, free-tier-aware AWS lab plans. It contains no hardcoded credentials, no paid API calls, and no backend by default.

This document explains the security model and how to report issues.

---

## Security model

### No hardcoded AWS credentials

Cloud Lab Guardian contains **no AWS credentials of any kind** — no access keys, no secret keys, no session tokens, no account IDs.

Every generated plan explicitly warns users against hardcoding credentials and recommends:
1. **AWS CloudShell** — no install or credential setup needed; uses your Console session
2. **IAM Identity Center (SSO)** — short-lived temporary credentials via `aws configure sso`
3. **Long-term access keys** — explicitly labeled *⚠️ Advanced / Not Recommended* in every plan

### Do not paste root credentials

Never use your AWS root account credentials with any tool, including this one. Always create a day-to-day IAM user with least-privilege policies.

### Generated plans are educational

Every plan generated by Cloud Lab Guardian is an **educational starting point**. The app:
- Does not execute any AWS commands on your behalf
- Does not connect to your AWS account
- In local deterministic mode, does not store, transmit, or log your project idea

You are responsible for reviewing every command before running it in a real AWS account.

### Use CloudShell or temporary credentials

Before using any generated CLI commands:
1. Open **AWS CloudShell** in the Console (zero-setup, uses your Console session)
2. Or use `aws configure sso` (IAM Identity Center) for local short-lived credentials
3. Avoid using long-term access keys stored in `~/.aws/credentials` for learning labs

### Optional Lambda backend

If you deploy the optional Lambda Function URL backend:
- The project idea, skill level, budget, and region are sent to your configured Lambda endpoint
- Prefer `AWS_IAM` authentication for the Function URL
- A static frontend cannot call an `AWS_IAM` Function URL directly unless requests are signed or proxied
- Never put AWS credentials, API keys, or bearer tokens in frontend `VITE_` variables
- `AuthType: NONE` is public unauthenticated access and should be used only for throwaway demos
- Use Lambda resource policies or a custom authorizer when exposing shared endpoints
- Set reserved concurrency to limit blast radius (see [DEPLOYMENT.md](DEPLOYMENT.md))
- Restrict CORS and validate request body size in the Lambda handler
- Rotate or delete the Function URL when no longer needed

## Reporting security issues

If you find a security vulnerability — especially one that could expose credentials or mislead users into unsafe AWS configurations — please report it **privately**:

1. **Do not** open a public GitHub issue
2. Email the maintainer directly (see GitHub profile for contact)
3. Include a description of the issue and steps to reproduce
4. Allow reasonable time for a fix before public disclosure

We treat all security reports seriously and will respond within 5 business days.

## Scope

| In scope | Out of scope |
|---|---|
| Incorrect or misleading safety warnings | UI styling bugs |
| Commands that could cause unintended charges | Test failures on edge-case prompts |
| Hardcoded credential patterns in generated output | Feature requests |
| Lambda backend exposure risks | Grammar/copy issues |
